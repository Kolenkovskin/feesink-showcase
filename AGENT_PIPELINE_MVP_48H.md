# MVP-план на 48 часов: ПК ↔ Телефон ↔ Агент(ы)

## Цель
За 48 часов собрать минимально безопасный и работающий пайплайн, который:
1. Видит состояние целевой страницы (в рамках разрешённых источников).
2. Выполняет ограниченные действия на телефоне (tap/swipe/open app).
3. Сохраняет артефакты в выделенную папку `handoff`.
4. Передаёт артефакты между агентами и ведёт полный лог событий.

> Важно: не использовать обход защит контента/ограничений сайтов. Работать только в рамках правил сервисов и закона.

---

## Архитектура (минимум)
- **Orchestrator (ПК)** — принимает задачу, строит шаги, пишет журнал.
- **Phone Bridge (телефон)** — исполняет только allowlisted-команды.
- **Artifact Service** — единая папка обмена `handoff` + манифесты.
- **Policy Guard** — проверка правил перед каждым действием.
- **Alerting** — уведомления и аварийная пауза.

---

## 7 компонентов, которые нужно поднять
1. **Task Queue**: очередь задач (`queued/running/blocked/done/failed`).
2. **Executor**: исполняет команды только из allowlist.
3. **State Reader**: читает состояние страницы/экрана разрешёнными методами.
4. **Artifact Writer**: сохраняет файлы только в `handoff`.
5. **Policy Engine**: валидация доменов, путей, лимитов действий.
6. **Audit Logger**: структурные логи (JSONL).
7. **Notifier + Kill Switch**: оповещения и аварийная остановка.

---

## Минимальные политики (включить сразу)
- Разрешённые домены: только явный allowlist.
- Разрешённые приложения: только те, что нужны для пайплайна.
- Разрешённые пути записи: только `handoff/`.
- Запрет массовых действий: лимит команд в минуту.
- Рискованные операции (скачивание/изменение системных настроек): только с подтверждением.

---

## Формат артефактов
Каждый результат в `handoff/` должен содержать:
- `*.md` — краткий отчёт агента по шагу.
- `*.json` — метаданные (task_id, время, источник, hash).
- `*.txt` или `*.html` — извлечённый контент (если разрешено политикой).

Пример имени:
- `2026-04-14_task-001_page-summary.md`
- `2026-04-14_task-001_meta.json`

---

## 10 событий, которые обязательно логировать
1. `task_created`
2. `policy_check_passed`
3. `policy_check_failed`
4. `command_sent`
5. `command_result`
6. `artifact_written`
7. `artifact_transfer_started`
8. `artifact_transfer_done`
9. `alert_triggered`
10. `killswitch_activated`

Поля для каждого события:
- `timestamp_utc`
- `task_id`
- `actor` (agent/service/user)
- `device` (pc/phone)
- `action`
- `result`
- `reason`

---

## 5 автоправил (safe automation)
1. **Unknown domain → block**
2. **Write outside `handoff/` → block**
3. **>N команд за 60 сек → pause + alert**
4. **Repeated failures (>=3) → fallback to manual approval**
5. **Any privileged action → require explicit confirmation**

## План внедрения на 48 часов

### День 1 (0–24ч)
- [ ] Поднять `Task Queue`, `Executor`, `Audit Logger`.
- [ ] Настроить `handoff/` и формат артефактов.
- [ ] Реализовать 3 базовых команды: `open_app`, `tap`, `swipe`.
- [ ] Включить `Policy Engine` с 3 правилами: домены, пути, лимит действий.
- [ ] Проверить 5 тест-сценариев (включая 2 негативных).

**Критерий успеха D1:**
- Задача проходит end-to-end и оставляет полный след в логах + артефакты в `handoff/`.

### День 2 (24–48ч)
- [ ] Добавить `Notifier` и `Kill Switch`.
- [ ] Добавить авто-паузы на аномалии.
- [ ] Добавить подтверждение на рискованные операции.
- [ ] Подключить второго агента только к чтению артефактов.
- [ ] Прогнать 10 сценариев + smoke test каждые 2 часа.

**Критерий успеха D2:**
- Система выдерживает серию задач, корректно блокирует нарушения и не пишет вне `handoff/`.

---

## Smoke test (минимальный)
1. Создать задачу `task-001`.
2. Проверить policy-pass.
3. Выполнить 2–3 команды на телефоне.
4. Сохранить отчёт и метаданные в `handoff/`.
5. Проверить события в логе.
6. Принудительно вызвать нарушение (неразрешённый путь) — убедиться, что блокируется и шлёт alert.

---

## Что НЕ делать на старте
- Не давать root/full-device доступ всем агентам.
- Не открывать запись во все папки телефона.
- Не отключать логи/уведомления «ради скорости».
- Не смешивать dev и личные аккаунты/данные.

---

## Быстрый чек-лист готовности
- [ ] Есть allowlist доменов и команд.
- [ ] Есть папка `handoff/` и только она для записи.
- [ ] Есть JSONL-аудит на каждое действие.
- [ ] Есть kill switch (локальный и удалённый).
- [ ] Есть сценарий восстановления после аварийной паузы.
